# GOST 34 Methodology Content Draft

## Methodology ID
`gost34`

## Overview

### What GOST 34 is
GOST 34 is a standards-based staged approach for creating automated systems, centered on formal requirements, documented lifecycle stages, technical assignment, project documentation, commissioning, and acceptance. In this project, it represents the most governance-heavy and regulation-oriented methodology in the supported set.

### Core idea
Instead of optimizing primarily for speed, sprint rhythm, or lightweight flow, GOST 34 organizes development through formalized stages, defined deliverables, and controlled acceptance. The process is especially relevant where regulation, procurement rules, contractual documentation, or institutional accountability are dominant.

### Why it matters
GOST 34 is important because it captures a delivery context that the other methodologies do not fully represent: projects where documentation, stage approval, and formal acceptance are not optional overhead, but core requirements of the environment. The standard explicitly treats the technical assignment as the key document defining requirements and creation order for an automated system, and it specifies staged creation logic for such systems. fileciteturn15file10 fileciteturn15file19

## First steps

### Suggested title
How to start applying GOST 34

### Step 1 — Examine the object of automation and formal need
Study the target organization or system context and justify why an automated system is needed. In the classic staged logic, this precedes detailed design work and frames the creation effort. fileciteturn15file19

### Step 2 — Form user requirements and concept
Gather the user’s formal requirements, study the object, and develop the system concept. This stage is meant to establish what kind of system is being created and on what basis. fileciteturn15file19

### Step 3 — Prepare and approve the technical assignment
Develop the formal technical assignment (ТЗ), which becomes the main document defining the requirements and creation procedure for the automated system. In the modern standard, the technical assignment is explicitly named as the core governing document. fileciteturn15file10

### Step 4 — Develop project documentation by stage
Prepare the necessary design documentation for the relevant project stages, such as preliminary decisions, technical project materials, and working documentation, depending on how the implementation is structured. fileciteturn15file19

### Step 5 — Prepare the object and personnel for commissioning
Before the system is considered operational, the organization, environment, equipment, and personnel must be prepared for introduction into use. Commissioning is not treated as a casual final step. fileciteturn15file19

### Step 6 — Conduct testing, trial use, and formal acceptance
Run preliminary tests, trial operation where required, and formal acceptance activities. GOST-style delivery logic depends heavily on explicit verification and acceptance before the system is considered introduced. fileciteturn15file19

### Practical note
GOST 34 should not be presented as just “more documents.” Its real logic is that system creation is governed through formal artifacts, stage outputs, and acceptance readiness, because the organizational environment requires that level of control.

## Core elements

### Suggested title
Stages, technical assignment, and acceptance logic

### Intro
GOST 34 is built around a staged system-creation process. The method’s meaning comes from formal stages, technical assignment, project documentation, commissioning, and acceptance, not from lightweight team routines.

### Group 1 — Stages

#### Formation of requirements
The process starts with examining the object of automation, justifying the need for the system, and forming user requirements. fileciteturn15file19

#### Development of concept
The future system concept is studied and elaborated before deeper project work proceeds. fileciteturn15file19

#### Technical assignment
The technical assignment formalizes the system requirements and creation procedure and acts as a controlling baseline. fileciteturn15file10

#### Project documentation stages
The classic staged structure includes design-oriented stages such as preliminary project decisions, technical project, and working documentation, with adaptation possible depending on the case. fileciteturn15file19

#### Commissioning and support
The lifecycle includes preparation for operation, testing, trial use, acceptance, and later support obligations. fileciteturn15file19

### Group 2 — Technical assignment and documents

#### Technical assignment (TA)
The core governing document that defines what is being created, why, and under what requirements and procedures. The 2020 standard explicitly frames it as the main document for creation and acceptance. fileciteturn15file10

#### Design and working documentation
Documentation is not supplemental decoration. It is part of how the system is defined, transferred, checked, and accepted.

#### Supporting materials for related parts
The standards-based logic allows linked documentation for components, software parts, technical means, and associated project work. fileciteturn15file10

### Group 3 — Testing and acceptance

#### Preliminary testing
The system is checked before broader operational introduction. fileciteturn15file19

#### Trial operation
Where relevant, the system is run in an operationally realistic mode before final acceptance. fileciteturn15file19

#### Acceptance testing
Formal acceptance is a major control point, not a symbolic closure step. fileciteturn15file19

#### Post-acceptance support
Support obligations continue after introduction into operation. fileciteturn15file19

## Team needs

### Suggested title
What GOST 34 needs from the team and organization

- Ability to work with formal requirements and approved baselines
- Documentation discipline strong enough for stage-based control
- Stakeholder structure that supports formal reviews and approvals
- Readiness to treat testing and acceptance as explicit governance mechanisms
- Coordination between business, technical, and organizational participants
- Capacity to prepare the operational environment and personnel before rollout
- Organizational need for traceability, accountability, and contractual clarity

### Practical note
GOST 34 is not well suited to teams that want informal, lightweight process control. It assumes that formality is part of delivery value because the surrounding institutional context requires it.

## Common mistakes

### Suggested title
What usually goes wrong

- Treating GOST 34 as just “Waterfall with more paperwork”
- Copying stage names without respecting the acceptance logic behind them
- Producing documentation mechanically without using it to control decisions
- Weak technical assignment that does not truly define scope and requirements
- Underestimating preparation of the object of automation and personnel
- Trying to force highly volatile exploratory product work into a rigid formal structure
- Delaying operational readiness issues until the end
- Reducing acceptance to a ceremonial sign-off instead of a real control milestone
- Ignoring that regulation and governance, not team preference, justify the method

### Why these mistakes matter
When teams keep only the visible bureaucracy and lose the governance logic, GOST 34 becomes expensive without becoming useful. Its value depends on formal coordination, controlled requirements, and acceptance discipline.

## Applicability

### Good fit when
- The project operates under formal regulation or institutional standards
- Documentation and approval are mandatory, not optional
- The customer requires a formal technical assignment and stage-based delivery
- Acceptance procedures are central to the project
- The system is part of an enterprise, public-sector, defense, industrial, or other formalized environment
- Traceability and accountability matter more than lightweight adaptation
- The organization can sustain structured coordination across lifecycle stages

### Weaker fit when
- Requirements are expected to evolve rapidly during delivery
- The project depends on frequent reprioritization and short adaptive cycles
- The team needs lightweight experimentation over formal staged control
- Work behaves like continuous service flow or product maintenance
- The environment does not actually require strong formalization
- The organization cannot sustain document and approval discipline
- Innovation risk and discovery dominate more than regulated execution

### Practical note
GOST 34 is strongest when formalization is externally justified by governance, regulation, or contract structure. Without that context, its overhead can outweigh its benefits.
